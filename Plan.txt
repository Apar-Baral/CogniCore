### Portable install (any Hermes on Windows or Linux)

One installer targets **Hermes’s own venv**, not a second agent:

| Platform | Command | Effect |
|----------|---------|--------|
| Windows | `scripts/install-hermes-cognition.ps1` | `pip install` into `%LOCALAPPDATA%\hermes\hermes-agent\venv` |
| Linux/macOS/WSL | `scripts/install-hermes-cognition.sh` | `pip install` into `~/.hermes/hermes-agent/venv` (or detected `hermes` venv) |
| Any (manual) | `hermes-agent-venv/bin/pip install hermes-cognition` | Same result |

**Distribution shape:**

```toml
# hermes-cognition/pyproject.toml
[project]
dependencies = ["cognition-engine>=0.3.54"]

[project.entry-points."hermes_agent.plugins"]
cognition = "hermes_cognition:register"
```

After install: enable in [config.yaml](C:\Users\lenovo\AppData\Local\hermes\config.yaml):

```yaml
plugins:
  enabled:
    - cognition
cognition:
  enabled: true
  data_dir: auto          # .cognition/ primary, .hermes/cognition/ fallback
  shield: { mode: medium, block_writes: true }
  budget: { session_tokens: 200000, zones: true }
  bootstrap: { inject_on_session_start: true, max_tokens: 2000 }
  semantic_index: false   # opt-in Chroma (~4GB)
```

**Project init (once per repo):** `hermes cognition init` → writes DNA + creates `.cognition/`; if `.hermes/cognition/` exists, migrate/merge per your choice (**both paths supported**).

## Feature combination clusters (build and wire together)

These clusters share data and must be implemented in the same phase to avoid broken handoffs.

### Cluster A — Project brain (features 1–6, 19–23, 28)

**Flow:** `init` → scan (#28) → DNA (#1) → plan (#24–25) → `on_session_start` → bootstrap (#19–21, #23) → work → `on_session_end` → handoff (#22) + avoid updates (#6).

```mermaid
sequenceDiagram
  participant User
  participant Hermes
  participant Bridge
  participant DNA as dna.json
  participant Boot as bootstrap.md

  User->>Hermes: new session
  Hermes->>Bridge: on_session_start
  Bridge->>DNA: load phase subtasks avoid
  Bridge->>Boot: ContextCompiler 3-tier
  Bridge->>Hermes: inject bootstrap into prompt
  User->>Hermes: work session
  Hermes->>Bridge: on_session_end
  Bridge->>DNA: session summary handoff metrics
  Bridge->>Boot: precompile next scenarios
```

**Data path resolution (your choice — both supported):**

1. If `./.cognition/dna.json` exists → use `.cognition/`
2. Else if `./.hermes/cognition/dna.json` → use `.hermes/cognition/`
3. On `init`, create `.cognition/`; if legacy `.hermes/cognition/` only, copy → `.cognition/` once (idempotent migration flag in DNA metadata)

### Cluster B — Shield loop (features 6–12, 23)

**Flow:** `pre_tool_call` on `write_file` / `patch` / `file_edit` → validate (#7–10) → block or auto-correct (#11) → on failure, append to avoid registry (#6) → stats (#12) → next bootstrap warns (#23).

### Cluster C — Budget + continuity (features 13–18, 15, 22)

**Flow:** Every API call → count tokens (#13) → zone check (#14) → at 90% inject wrap-up (#15) → session end writes handoff consumed next start (#22) → efficiency + usage (#17–18) stored in DNA.

**Hook mapping:** `pre_llm_call` (block/warn), `post_api_request` (metering), `on_session_end` (scores).

### Cluster D — Planning + adaptation (features 24–28, 5, 26–27)

**Flow:** User adds feature → impact (#27) → deviation strategy (#26) → update feature registry (#5) → reorder phases (#25).

### Cluster E — Visualization (features 29–34)

Read-only over DNA + session metrics; expose via `hermes cognition status [--detailed|--phase]` and `/status`.

### Cluster F — Learning flywheel (features 35–41, 12, 37)

**Flow:** Session end → synthesizer (#35–36) → trends (#41) → recommendations (#40) → RL update (#38–39) → calibration (#37) → DNA insights → optional inject in bootstrap.

### Cluster G — Cross-project transfer (features 42–45) — new code

**Flow:** Project complete → extract patterns (#43) → global registry (#42) → new `init` → similarity (#44) → plan bootstrap (#45).

Depends on Cluster A DNA schema; store global data under `~/.cognition/` (OS-neutral via `pathlib`).

### Cluster H — Multi-agent dev (features 46–49) — new code

**Flow:** Phase sub-tasks (#3) → decomposer (#47) → `delegate_task` with role-specific prompts (#46) → result merger (#48) → conflict resolver (#49).

Uses Hermes native delegation; plugin supplies prompts, contracts, and merge tooling — not a second agent runtime.

### Cluster I — Model intelligence (features 50–54)

**Flow:** Task complexity estimate → router (#51) → circuit breaker on failures (#52) → normalized provider calls (#53) → cost compare UI (#54).

Integrate with existing [config.yaml](C:\Users\lenovo\AppData\Local\hermes\config.yaml) `model.default` / `providers` — router suggests override in `pre_llm_call` or slash `/model recommend`.

## Cross-platform portability checklist

- All paths via `pathlib.Path`; no hardcoded `C:\` or `/home`
- Hermes home: use Hermes’s `get_hermes_home()` when integrating config; CE global config stays `~/.cognition/` (or `%USERPROFILE%\.cognition` on Windows)
- Install script detects venv: `where hermes` → dirname → `..\venv\Scripts\python` (Win) or `bin/python` (Unix)
- Optional extras: `pip install cognition-engine[semantic]` only when Chroma needed
- No dependency on CE Textual TUI (`repl/`) — Hermes remains the only chat UI
- Version pin: `cognition-engine==0.3.54` until facade API stabilizes
